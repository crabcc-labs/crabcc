# crabcc v4.0 — Data Layer 2.0

**Status:** Design approved 2026-05-16 (brainstorming session, this branch).
**Author:** Peter Lodri (driver) + Claude (drafting).
**Timebox:** Weekend hackathon (~2 days wall-clock).
**Release target:** v4.0.0 (major bump from 3.2.0).

## 1 — Theme

v4.0 replaces the current name-based edge graph with a **symbol-ID-keyed graph
model** and exposes a real knowledge-graph CLI surface on top. One
major-version-justifying breaking change (the schema) + one new user-visible
capability (graph queries that actually traverse a graph, not a name index).

The current `edges(src_file_id, src_symbol TEXT, dst_name TEXT, kind, line)`
table forces every query through approximate `dst_name` matching. The
`bare_name()` qualifier-stripping helper shipped in v3.2 (commit `8197938`)
is a symptom of this — it papers over the loss of resolution, but call sites
to `Foo::open` and `Bar::open` collapse into one logical destination, breaking
transitive walks. KG ops (blast-radius, why-path, hot-symbols, importers)
require unambiguous identity.

## 3 — Schema (breaking)

```sql
-- schema/001_init.sql, v4.

CREATE TABLE files (...)             -- unchanged

CREATE TABLE symbols (
    id            INTEGER PRIMARY KEY,
    file_id       INTEGER NOT NULL REFERENCES files(id) ON DELETE CASCADE,
    name          TEXT NOT NULL,
    qualified     TEXT,                 -- "crate::module::Foo" when extractable
    kind          TEXT NOT NULL,
    parent_id     INTEGER REFERENCES symbols(id),
    line_start    INTEGER NOT NULL,
    line_end      INTEGER NOT NULL,
    signature     TEXT,
    signature_enc INTEGER NOT NULL DEFAULT 0,
    visibility    TEXT
);
CREATE INDEX idx_symbols_name ON symbols(name);
CREATE INDEX idx_symbols_file ON symbols(file_id);
CREATE INDEX idx_symbols_qual ON symbols(qualified);

CREATE TABLE edges (
    id              INTEGER PRIMARY KEY,
    src_symbol_id   INTEGER NOT NULL REFERENCES symbols(id) ON DELETE CASCADE,
    dst_symbol_id   INTEGER NOT NULL REFERENCES symbols(id) ON DELETE CASCADE,
    kind            TEXT NOT NULL CHECK (kind IN ('call','ref','import','inherit','impl')),
    line            INTEGER NOT NULL
);
CREATE INDEX idx_edges_dst       ON edges(dst_symbol_id);
CREATE INDEX idx_edges_src       ON edges(src_symbol_id);
CREATE INDEX idx_edges_dst_kind  ON edges(dst_symbol_id, kind);

-- Sentinel-symbol pattern for unresolved edges (Ruby/Java/Swift, or genuinely
-- ambiguous calls). One sentinel row per unique unresolved name; queries that
-- want recall can union edges through these.
CREATE TABLE unresolved_names (
    symbol_id INTEGER PRIMARY KEY REFERENCES symbols(id) ON DELETE CASCADE,
    name      TEXT NOT NULL UNIQUE
);
CREATE INDEX idx_unresolved_name ON unresolved_names(name);

INSERT OR IGNORE INTO meta(key, value) VALUES ('schema_version', '4');
```

The pre-v4 `edges(src_file_id, src_symbol TEXT, dst_name TEXT, kind, line)`
shape is **dropped** entirely. No migrator. The v3.2-introduced auto-wipe
path (extended to detect `schema_version < 4 || schema_v4_built != '1'`)
rebuilds the index on first open.

### Resolver contract

```rust
pub struct SymbolId(pub i64);

pub struct ScopeCtx<'a> {
    pub file_id: i64,
    pub current_module: Option<&'a str>,
    pub imports: &'a [ImportSpec],
    pub local_defs: &'a HashMap<&'a str, SymbolId>,
}

pub trait Resolver: Send + Sync {
    /// Resolve a use-site name (`Foo` in `let x: Foo = ...`) to a definition.
    /// None => emit an unresolved-name edge via the sentinel pattern.
    fn resolve_ref(&self, scope: &ScopeCtx, name: &str) -> Option<SymbolId>;

    /// Resolve a call-site callee (`foo()` / `obj.method()` / `Type::assoc()`).
    fn resolve_call(&self, scope: &ScopeCtx, callee: &str) -> Option<SymbolId>;
}
```

Per-language modules implement `Resolver` with their own scope-walking logic.
Languages without a resolver fall back to a `NameOnlyResolver` that returns
`None` (always sentinel) — old name-based `lookup refs`/`lookup callers`
queries continue to work via the `unresolved_names` side table.

### Extractor (two-pass)

```rust
// pseudo-Rust
fn extract_file(path: &Path, store: &Store, resolver: &dyn Resolver) -> Result<()> {
    let file_id = store.upsert_file(...)?;
    // Pass 1: walk every definition node, INSERT into symbols, build local_defs map.
    let local_defs = collect_defs(tree, file_id, store)?;
    // Pass 2: walk every use/call node, ask the resolver, INSERT into edges.
    emit_edges(tree, file_id, store, resolver, &ScopeCtx { ..., local_defs })?;
    Ok(())
}
```

### Migration

`Store::needs_reindex` becomes:

```rust
let has_files = /* count(*) > 0 */;
let schema_v4_built = meta.get("schema_v4_built") == Some("1");
let needs_reindex = has_files && !schema_v4_built;
```

`mark_ref_edges_built()` is renamed to `mark_schema_v4_built()` and writes
`schema_v4_built=1`. The CLI auto-wipe path (in `main.rs`, lines ~1115 of
the v3.2 state) is updated to match.

## 5 — New CLI surface

```
crabcc graph blast-radius <symbol> [--depth N] [--kind call|ref|all]
crabcc graph why <src-symbol> <dst-symbol> [--max-depth N]
crabcc graph hot-symbols [--top N] [--kind call|ref|all]
crabcc graph importers <file-or-module> [--depth N]
```

Existing commands (`graph walk`, `graph cycles`, `graph orphans`) keep their
flag shape but their output now references symbol-IDs (and qualified names)
rather than raw destination strings.

## 6 — Wave dispatch (orchestrator)

15 tasks, three waves. Model rotation: each task is assigned
`models[task_id % len(models)]` over the four-model rotation
`[tencent/hy3-preview, deepseek/deepseek-v4-flash, deepseek/deepseek-v3.2,
deepseek/deepseek-v4-pro]`. Concurrency cap raised to 10 per user
authorization 2026-05-16.

| 2 | `Store` API: `insert_symbol`, `insert_edge_resolved`, `mark_schema_v4_built`, updated `needs_reindex` | W1 |
| 3 | `resolve.rs`: Resolver trait + ScopeCtx + sentinel helpers + NameOnlyResolver | W1 |
| 4 | `extract.rs`: split into two-pass, route through Resolver | W1 |
| 5 | Rust resolver (`extract/resolve_rust.rs`) | W2a |
| 6 | TS/JS resolver (`extract/resolve_ts.rs`) | W2a |
| 7 | Python resolver (`extract/resolve_python.rs`) | W2a |
| 8 | KG: `query/blast_radius.rs` | W2b |
| 9 | KG: `query/why.rs` | W2b |
| 10 | KG: `query/hot_symbols.rs` | W2b |
| 11 | KG: `query/importers.rs` | W2b |
| 12 | CLI dispatch: wire the 4 query modules into `crabcc graph` subcommands; upgrade existing `walk/cycles/orphans` | W2c |

crabcc-core -p crabcc-cli` must pass before the next wave dispatches.

## 8 — Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Resolver ambiguity (multiple defs share a name) | High | Return ALL candidates from the resolver; queries union. Don't pick one silently. |
| Two-pass extraction regression on indexing wall-clock | Med | Pass 1 + Pass 2 are each ~equal to today's single pass; expect ≤20% slower. Measured against S4. |
| Hackathon coder agents misuse resolver scope walker | Med | Each per-language resolver prompt inlines the exact tree-sitter query strings + the `ScopeCtx` definition. No "design as you go". |
| Wave 2 KG ops collide on `query.rs` | Cert. | Each KG op gets its own file (`query/blast_radius.rs`, etc.). CLI integration is a separate W2c task. |
| LSP/MCP regression | Med | W3 integration test gates merge. |

## 9 — Migration UX

No CLI flag, no migrator, no user choice.

First v4 open on a pre-v4 index:
```
crabcc: index built with schema v3; wiping and re-indexing for symbol-ID edges...
```

Then the index runs to completion. Wall-clock matches a fresh index (≤60s on
this 13k-file repo). Users were conditioned to this contract in v3.2 — they
already saw the same auto-wipe message when upgrading from v3.1 indexes
without `ref_edges_built`.
